エッセイ — 03

X年前のリサーチフレームワークが、いまAIの「指示書」を書いている。ほとんどのチームはその存在を知らない。

フレームワークは30年前から存在していた。その「出番」が、いま来た。

2025

マクドナルドの朝。通勤途中の男性が、ミルクシェイクを買う。

空腹だからではない。

長い退屈な通勤を少しだけ楽しくしたい。片手で持てる。時間をかけて飲める。昼まで腹が持つ。朝の自分への小さなご褒美。

ジョブは「ミルクシェイクを飲む」ではなかった。ジョブは「退屈な通勤を生き延びて、自分にちょっといいことをした気分で職場に着く」だった。

この発見が、マクドナルドの商品開発の考え方を根本から変えた。

そしてこの「ジョブ」という発見の仕方が——30年の時を経て——いまAIの時代に、全く新しい重要性を帯びている。

JTBDの本当の起源——ほとんどの人が知らない事実

Jobs-to-be-Doneは新しいアイデアではない。

そして、それを本当に生み出した人物は、多くの人が思っている人物ではない。

Tony Ulwick——Strategynの創設者であり、Outcome-Driven Innovation（ODI）の考案者——が、1980年代後半にCordis Corporationなどの企業と仕事をする中でコアコンセプトを開発した。彼はこれを、顧客の未充足ニーズをイノベーション戦略に接続するための、厳密で再現可能な方法論に仕上げ、ODIプロセスの特許を保有している。私は彼の仕事を深く研究してきた。人々が実際に必要としていること——言っていることでも、していることでもなく、その根底にある「前進しようとしていること」——を理解するためのフレームワークとして、最も精密で実用的なものの一つだ。

多くの人はClayton ChristensenがJTBDの創始者だと認識している。これは誤帰属だ。ChristensenはUlwickの仕事に関わり、その思考を自身のイノベーション理論に取り込み、特に有名なミルクシェイク・マーケティングの話と後の著書 Competing Against Luck（2016年）を通じて大きく普及させた。この思考を広めた彼の貢献は本物であり重要だ。しかし起源はUlwickにある。

両者が指し示すコアインサイトは同じだ——人は製品を「買う」のではなく、人生の中のジョブを遂行するために「雇う」。このインサイトは数十年にわたってプロダクトデザインの語彙の一部であり続けてきた。経験豊富なデザイナーの多くはこれに出会っている。多くが使ってきた。

では、なぜ私はいま、緊急であるかのようにこれについて書いているのか？

その歴史のほとんどの期間、JTBDはリサーチのインプットだった——ユーザーを理解する方法であり、その理解はインターフェースデザインに翻訳されていた。有用で、時に変革的だったが、プロダクトそのものから一歩離れていた。

AIの時代、その一歩が消える。JTBDはリサーチのインプットであることをやめ、システム設計言語になる。ジョブの理解とシステムの定義の間のギャップが、ほぼ完全に閉じる。

だからいまより重要なのだ。フレームワークが変わったからではない。それを使って何をするかが変わったからだ。

Jobs-to-be-Doneとは実際に何か

初めて触れる人のために。Jobs-to-be-Doneフレームワークは、ユーザーはあなたのプロダクトを欲しがっているのではなく、人生の特定の状況で前進を遂げたいのだと言う。その前進を助けるために、あなたのプロダクトを「雇う」のだ。

ジョブには3つの構成要素がある：

機能的ジョブ： ユーザーが文字通り達成しようとしていること。フライトを予約する。レポートを書く。プロジェクトを整理する。

感情的ジョブ： それをしている間、またはその結果として、ユーザーがどう感じたいか。自信がある。コントロールできている。他者に印象を与える。安堵する。

社会的ジョブ： その結果として、ユーザーが他者からどう見られたいか。有能だ。思慮深い。良い親だ。信頼できる同僚だ。

3つとも現実だ。3つとも行動に影響する。機能的ジョブだけのためにデザインすることは、技術的に正しいが誰にも愛されないものを作る最も一般的な方法だ。

先ほどのミルクシェイクの例がそれを証明している。ジョブは「ミルクシェイクを消費する」ではなかった。ジョブは「退屈な通勤を生き延びて、自分にちょっといいことをした気分で職場に着く」だった。

2.0時代、JTBDはなぜ重要だったか

UX 2.0時代、JTBDリサーチはより良いインターフェースを生み出した。

ユーザーがアプリを「タスク完了」のために雇っているのではなく、「混沌とした状況のコントロール感」のために雇っていると理解したなら、デザインは変わった。スピードより自信。機能より明快さ。システムにできることより、ユーザーが理解し信頼できること。

これは価値があった。デザインを「フローを動かす」から「フローの背後にある人間の目標に仕える」へと引き上げた。

しかし、常に翻訳ステップがあった。

```

ジョブを理解する

→ インターフェースにとってそれが何を意味するか決める

→ インターフェースをデザインする

→ ユーザーがそれを通じてジョブを達成することを祈る

```

ジョブがデザインに情報を与えた。しかしユーザーがまだ仕事をした。

3.0時代に変わること——翻訳ステップの圧縮

AIがユーザーに代わって行動するとき、翻訳ステップは劇的に圧縮される。

AIはジョブを知る必要がある——デザイン決定のインスピレーションとしてではなく、動作指示として。ジョブの構造がシステムの意思決定フレームワークそのものになる。

具体例で見せる。

シナリオ： ユーザーがAIアシスタントに、仕事量についてマネージャーとの難しい会話の準備を手伝ってほしいと頼む。

表面のリクエスト： 「ミーティングの準備を手伝って。」

実際のジョブの分解：

機能的： 会話がまずい方向に行かずに済むこと。明確にコミュニケーションすること。解決策にたどり着くこと。

感情的： 不安ではなく準備ができていると感じること。聞いてもらえたと感じること。愚痴を言っているのではなく問題解決をしていると感じること。

社会的： うまく上方管理できる有能なプロフェッショナルとして見られること。仕事量をさばけない人間としてではなく。

成功基準： 過負荷に対処する具体的な計画か、現状がなぜ必要でいつまで続くかの明確な理解を持って、ミーティングを終えること。

コントロールしたいこと： 具体的な発言内容。問題のフレーミング。

委任して構わないこと： この種の会話に適したフレームワークの発見。マネージャーの反応の予測。論点の構造化。

では、このジョブ構造があるときとないときで、AIが何をできるか見てみよう。

ジョブ構造なし： 「難しい職場の会話のためのヒントです。1. 問題を具体的に。2. 解決策に焦点を...」

汎用的。おそらく使える。しかしこの人の実際の状況のためにデザインされていない。

ジョブ構造あり： AIは、この人が準備ができていてプロフェッショナルだと感じたい——愚痴を言っているようには感じたくない——と知っている。成功とは計画か説明を持って終えることだと知っている。具体的な言葉遣いは自分で決めたいが、構造と予測の助けは欲しいと知っている。

いまやAIは、会話を苦情ではなく問題解決としてフレーミングする手助けができる。マネージャーの起こりうる反応を予測できる。防衛的に聞こえずに押し返す必要がある瞬間の準備ができる。

JTBDの理解はインターフェースに情報を与えたのではない。システムの動作指示そのものになった。

最も重要な4つの問い——標準を超える深さ

AIシステム設計に情報を与えるJTBDリサーチを行うとき、私は標準的なフレームワークよりも深い4つの問いに焦点を当てる。

1. 成功は「見た目」ではなく「感覚」としてどう見えるか？

機能的な成功基準は簡単だ。フライトが予約された、レポートが書かれた、ミーティングがスケジュールされた。しかし純粋に機能的成功を最適化するAIシステムは、しばしば空虚に感じられる。

重要な問い：このジョブがうまく行われたとき、ユーザーは何を感じるか。何をもはや心配しなくなるか。以前はできなかった何ができるようになるか。

これはAIのコミュニケーション方法を形作る——何をするかだけでなく。フライトを予約して「完了」と言うAIと、「会議の2時間前に到着でき、夜10時までに帰宅できる便が見つかりました。予約しますか？」と言うAIでは、同じ機能的ジョブに全く異なる形で仕えている。

2. ユーザーは何のコントロールを手放したくないか？

これはオートノミー設計——AIがどこまでやり、どこから人間の判断に委ねるか——に最も直接的に影響する問いだ。

ほぼすべてのジョブには、ユーザーが自分で所有したい部分がある。重要なメールの具体的な言葉遣い。大きな購入の最終判断。子どもに何を伝えるかの決断。

この理解はUXの好みの問題だけではない。「委任する価値のあるジョブ」と「自分で所有する価値のある決断」の境界を尊重することだ。これを間違えること——ユーザーがコントロールしたかったことをAIがやってしまうこと——は、信頼を壊す最速の方法の一つだ。

3. 技術的に正しくても「間違っている」と感じるのはどんなときか？

この問いは、機能的分析が見落とす感情的・社会的次元のジョブを浮かび上がらせる。

最安値のオプションを予約する旅行AIは技術的に正しいかもしれない。しかしユーザーがAIを雇った理由の一部が、賢く考慮された選択をしたと感じること——ただ最も安いだけではなく——だったなら、技術的に正しいアウトカムが間違った感情を生む。

この問いへの答えは、しばしばAIの振る舞いの制約条件になる：「トレードオフを提示せずに純粋に価格だけで最適化するな。」

4. どんな文脈がジョブを根本から変えるか？

同じユーザー、同じリクエストでも、文脈次第で全く異なるジョブになりうる。

「チームへのメッセージのドラフトを手伝って」——普通の月曜日なら生産性のジョブだ。同じリクエストが、大きな組織変更の翌日には感情的リーダーシップのジョブになる——ユーザーは明確にコミュニケーションしながら、自分自身の感情とチームの不確実性も管理する必要がある。

文脈のシグナルを検出し、ジョブの理解を調整できるAIシステムは、同じリクエストのすべてのインスタンスを同一に扱うシステムより、劇的に有用だ。

AIのためのJTBDリサーチは何が違うか——同じインタビュー、異なるアウトプット

インタビューは同じに見える。分析が異なる。

標準的なJTBDリサーチセッションでは、ユーザーの苦闘の瞬間と、遂げようとしている前進に耳を傾ける。問う：これが必要だと最初に気づいたとき、何をしていたか。それ以前に何を試したか。何がやめようとさせかけたか。

AIのためにデザインするとき、同じことに耳を傾けるが、同時に、運用可能にできるほど具体的なジョブのモデルを構築している。

リサーチの後、ペルソナドキュメントやジャーニーマップを作る代わりに——あるいはそれらに加えて——生み出すのはこれだ：

ジョブ・ステートメント： ユーザーが達成しようとしているジョブの明確で構造化された記述。フォーマットは「[状況]のとき、[動機]をしたい、それにより[成果]を得る」。

成功基準： 「うまくいった」がどう見えるかの具体的でテスト可能な定義——機能的、感情的、社会的に。

オートノミー・マップ： ジョブのどの部分をユーザーが委任したく、どの部分を自分で所有したいかの分解図。

コンテキスト・トリガー： ジョブがシフトしたことを示すシグナル——同じ表面のリクエストが異なる根底のニーズを表していること。

これらの成果物はAIシステム設計に直接供給される。インスピレーションではない。仕様書だ。

より深い論点——フレームワークは同じ、ステークスが上がった

Jobs-to-be-Doneは常に、一見シンプルなインサイトについてだった。人はあなたが思っていることをしていない。あなたのプロダクトを使っているのではない——人生の中で前進しようとしていて、あなたのプロダクトはそのために使おうとしている一つの手段にすぎない。

UlwickがStrategynで構築したもの——そして私が実践で適用してきたもの——は、このインサイトを運用可能にする方法だ。マインドセットの転換だけではなく、未充足ニーズを、実際のプロダクト決定を駆動できるほどの具体性で浮かび上がらせる厳密なプロセス。

このインサイトに基づいて行動するには、常に表面レベルのタスク分析以上のものがデザイナーに求められてきた。人間の動機への本物の好奇心、曖昧さに留まる意志、ユーザーが直接言語化できる以下のレベルで行動のモデルを構築するスキル。

それは何も変わらない。変わるのは、構築したモデルが自分の頭の中やリサーチドキュメントにとどまらないということだ。システムに入る。 AIの行動を形作る。自律型システムがリクエストそのものだけでなく、リクエストの背後にいる人間に仕えるかどうかを決定する。

だからJTBDはいまより重要なのだ。フレームワークは同じ。それをうまく適用するステークスが上がった。

オリジナルの方法論を深く知りたいなら、Tony UlwickのStrategynでの仕事から始めるべきだ。この思考の最も厳密なバージョンがそこにある。

この記事は、[インターフェースは「盾」だった。AIがそれを取り去った。→]のコンパニオンピースだ。そこで「意図＝システムパラメータ」というフレーミングが有用だと感じたなら、この記事はその背後にあるリサーチ手法をより深く掘り下げる。

AIプロダクトに取り組んでいて、自分の文脈で意図リサーチをどう構造化するか考えたいなら——その会話こそ、私が最も価値を見出すものだ。気軽に連絡してほしい。

P.S. 私自身の失敗について書いておく。以前、あるAIプロダクトのデザインで、私はユーザーの機能的ジョブだけを精密に定義した。「会議のスケジュールを最適化する」。AIは完璧に動いた。カレンダーの空きを見つけ、参加者の都合を調整し、最も効率的な時間を提案した。技術的に非の打ち所がなかった。しかしユーザーは使わなくなった。なぜか。そのユーザーにとって、会議のスケジューリングは「自分がチームを大切にしていると示す行為」だった。感情的ジョブと社会的ジョブを完全に見落としていた。AIが効率を最適化するほど、ユーザーは「冷たい上司」になった気がした。それ以来、私は機能的ジョブだけでは動作指示の半分にもならないと、身をもって知っている。
