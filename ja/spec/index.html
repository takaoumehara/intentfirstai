実践的な使い方

どう使うか

今日から始められる3つの活用法。

01

AIデザインレビュー

Token Specを任意のAI会話に貼り付ける。通知フロー、チェックアウト画面、オンボーディング体験をContext Grammarの観点で評価させる。AIは見落としを指摘する。「このフローは認知負荷を無視している。ユーザーが疲弊しているときはどうなる？」

02

データアーキテクチャ

brain-schema.yamlをユーザープロファイルシステムの出発点にする。3層モデル（Identity / Accumulated Learning / Right Now）は、安定した事実、学習されたパターン、今この瞬間の状態を分離する。フォークして、不要な部分を削り、必要な部分を拡張する。

03

条件分岐UIロジック

33のデザインルールは「Xのとき、Yする」の条件式。そのまま実装の参照仕様になる。すべてのルールが実際のシナリオに遡れる。認知負荷が枯渇した状態で帰宅中の運転、子どもがテレビを見ている横での買い物。

Quick Start

2分で試す

チュートリアルを選んで、AI会話を1つ開くだけ。インストール不要、セットアップ不要。YAMLファイルとプロンプトだけで始められる。

01

Context GrammarでUIをレビューする

デザイナー向け
  1. Claude.ai、ChatGPT、Geminiのいずれかを開く。どのAIチャットでも動く。
  2. context-tokens-spec.yamlを会話に添付する（ファイルをドラッグするか、中身を貼り付ける）。
  3. 自分のアプリ画面のスクリーンショットをアップロードするか、フローを数行で説明する。
  4. このプロンプトを送る: 
    Context Grammar Specに基づいて、この画面をレビューしてください。
考慮できていないTokenはどれですか？
ユーザーの認知負荷が高い場合、公共の場にいる場合、
小さい画面で見ている場合に何が壊れますか？
  5. AIの回答を読む。自分が設計時に見落としていた盲点を指摘してくれる。

AIの回答例: 「この決済フローはCognitive Loadを無視している。ユーザーが疲弊しているとき、12個の支払い方法は選択麻痺を引き起こす。よく使う3つに絞るべきだ。また、Social Exposureも見落としている。注文合計が大きなテキストで表示されるため、電車の中では隣の乗客から丸見えになる。」

自力でエッジケースを洗い出すと数時間かかる。Specを渡せば、AIが30秒で指摘する。
02

コンテキスト適応UIのバリエーションを生成する

開発者向け
  1. AIチャットを開き、design-rules.yamlを添付する。
  2. 自分の機能を2〜3文で説明する（例: 「キュー管理とソーシャル共有を持つ音楽プレイヤー」）。
  3. このプロンプトを送る: 
    Design Rules Specを使って、この機能の
UIバリエーションを3つ生成してください。

1. cognitive_load: high, physical_state: driving
2. social_exposure: public, form_factor: smartwatch
3. cognitive_load: low, autonomy_dial: Suggest

各バリエーションで、どのルールが適用されるかを説明し、
具体的なUI変更を示してください。
  4. 各コンテキストに対応した具体的なUI仕様が返ってくる。汎用的な提案ではなく、ルールに裏打ちされた適応設計。

AIの回答例: 「バリエーション1（運転中 + 疲弊）: ルールDR-003を適用。キューを『次の1曲』に折りたたむ。共有ボタンは完全に非表示。音声操作のみ。バリエーション2（公共 + スマートウォッチ）: ルールDR-017。パーソナルコンテンツを含むアルバムアートを非表示。再生コントロールのみ表示、ソーシャル機能なし。音声確認の代わりにハプティクス。」

1つのUIを作って「どこでも使えますように」と祈る代わりに、テスト済みルールに基づいたコンテキスト別バリエーションが手に入る。
03

ユーザーコンテキストモデルを構築する

エンジニア向け
  1. AIチャットを開き、brain-schema.yamlを添付する。
  2. 自分のアプリとユーザーを説明する（例: 「食事制限のある忙しい家族向けのレシピアプリ」）。
  3. このプロンプトを送る: 
    このBrain Schemaを私のレシピアプリ用に
カスタマイズしてください。

3つのメモリ層にそれぞれ何を入れるべきですか？
- Identity: 各ユーザーについてどんな安定した事実？
- Accumulated: 時間をかけて何を学習すべき？
- Right Now: レシピに関連する一時的な状態は？

また、「ディナーパーティー」シナリオの
Disposable Brainはどんな形になりますか？
  4. ドメインに特化したデータモデルが返ってくる。具体的なフィールド、型、ライフサイクルのルール付き。

AIの回答例: 「Identity層: アレルギー（家族メンバーごと）、キッチン器具リスト、料理の好み、スキルレベル。Accumulated層: 曜日別の調理頻度、食材の代用履歴、レシピの途中離脱パターン。Right Now: パントリーの在庫（直近の買い物レシートから）、今夜の調理時間、食事する人数。ディナーパーティーのDisposable Brain: ユーザーが『ディナーパーティーを計画する』と言った時に生成。ゲストの食事制限、コース計画、準備タイムラインを追跡。学習したホスティングパターンをAccumulatedに戻してから消滅する。」

データモデリングの議論に何週間も費やす代わりに、フォークできるテスト済みの構造が手に入る。白紙を見つめる必要はない。
